## fix
Bind transport sockets to physical interface IP, not just IP_UNICAST_IF
1.28.15 set IP_UNICAST_IF on the daemon's transport sockets but a profile from a 1.28.15 install showed the daemon's UDP packets were still entering bridgeInbound (5.6s in the self-bypass branch). Turns out IP_UNICAST_IF on connected UDP sockets is only "advisory" on some Windows builds — the kernel can still pick the TUN default route. Now we additionally bind the socket source IP via Dialer.LocalAddr to the physical interface address, which forces the kernel's source-address selection and routing decision. IP_UNICAST_IF is kept as belt-and-suspenders. Added a one-shot log line so the next bug shows local/remote/ifIndex of the actual socket.
