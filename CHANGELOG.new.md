## fix
UDP and TLS transports now use protected dial — bypass TUN routing
The proxy server transports were using net.DialUDP / tls.DialWithDialer directly, with no socket-level interface binding. Once the TUN engine added its default route, every outbound packet from the daemon's own proxy stream followed it — straight back into bridgeInbound. The new self-detection branch saw "this is our own packet" and reflected it through NAT bypass, doubling the work and burning CPU on a feedback loop. After the IP_UNICAST_IF byte-order fix in 1.28.14 the loop stopped reaching infinity, but the daemon still spent a lot of CPU on the bounce. Now both UDP and TLS transports go through protectedDialUDP, which binds the socket to the physical interface (Windows IP_UNICAST_IF / macOS IP_BOUND_IF). Loopback dials skip the binding so tests still pass.
