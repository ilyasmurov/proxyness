## fix
TUN engine wedged in Reconnecting after UDP→TLS fallback + D3 force-close
Triggered by the 1.29.0 server redeploy: client's D1 caught the server session close, reconnected, UDP timed out and AutoTransport fell back to TLS, then D3 fired on accumulated stream failures and force-closed the TLS transport. TLSTransport had no DoneChan method, so AutoTransport.DoneChan returned nil, and the healthLoop's `<-doneCh` branch blocked forever on a nil channel — D1 never got the wake-up it was supposed to. D2 just counts failures without reconnecting, so after 60 seconds (12 × 5s) it exhausted and stopped the engine, leaving the user to press Connect manually. Two fixes: (1) TLSTransport now implements DoneChan + Alive to match UDPTransport so AutoTransport delegation works uniformly; (2) D3 no longer relies on Close-then-DoneChan to trigger D1 — it calls reconnectTransport directly, mirroring D1's own recovery path.
