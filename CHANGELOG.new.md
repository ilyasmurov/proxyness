## fix
Windows IP_UNICAST_IF byte order — protectedDial was silently ignored, daemon traffic looped through TUN
Microsoft requires the interface index in network byte order via htonl. The previous code used `uint32(ifIndex) << 16` which gave the wrong value (`0x00030000` for index 3 instead of the correct `0x03000000`). Windows silently ignored the bogus socket option and the "bypass" sockets followed the system default route — which in TUN mode is the TUN device. Daemon's own outbound TCP looped back into bridgeInbound. "All traffic" mode survived because the main proxy stream is opened once and multiplexed; "Selected" mode broke immediately because every isSelf bypass needed a fresh dial. Fix is `<< 24`, equivalent to htonl on little-endian.
