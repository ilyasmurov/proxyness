# Proxy Server: SQLite → Postgres Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Перевести прокси-сервер (бинарь + admin UI в его составе) со встроенной SQLite на удалённый Postgres 16 на Aeza. Оба VPS (Aeza локально, Timeweb через WG туннель) начнут читать и писать в одну общую БД. Добавить 60s-TTL кеш перед `GetDeviceByKey` чтобы убрать 85ms penalty с hot path на Timeweb.

**Architecture:** Перепишем пакет `server/internal/db` с `modernc.org/sqlite` на `github.com/jackc/pgx/v5` (через `pgx/v5/stdlib` для совместимости с `database/sql`, где это упрощает порт). Вся публичная поверхность пакета (типы `User`/`Device`/`TrafficStat`, методы `CreateUser`, `GetDeviceByKey` и т.д.) остаётся — меняются только тела функций: плейсхолдеры `?` → `$1/$2/...`, `INTEGER` для bool → `BOOLEAN`, `INSERT OR IGNORE/REPLACE` → `ON CONFLICT`, `AUTOINCREMENT` уходит, inline `ALTER TABLE` миграции удаляются (всё живёт в `server/internal/db/pg/schema.sql`). Кеш — отдельный тип `deviceCache` с `sync.RWMutex` + `map[string]cacheEntry` + TTL; обёртка вокруг `GetDeviceByKey`. Инвалидация: admin-мутации девайсов вызывают `cache.Invalidate(key)` напрямую (админ и прокси — один процесс, доступ свободный). CLI меняет `-db /data/data.db` на чтение `PROXYNESS_DB_URL` из env.

**Tech Stack:** Postgres 16 (уже готов из plan 1+2), pgx/v5 (уже в go.mod из plan 2), Go workspace, Docker. Amnezia WG на Timeweb остаётся как fallback для пользователя на время раскатки.

**Prereqs:**
- Plan 1 + 2 выполнены: WG `wgpn0` живёт, Postgres на Aeza слушает 10.88.0.1, БД `proxyness` залита данными, `/etc/proxyness/db.env` на обоих VPS.
- На момент деплоя прода — пользователь переключен на Amnezia WG (независимо от Proxyness), чтобы рабочий VPN Claude'а не оборвался посреди раскатки.

**Out of scope:** Вынос админки в отдельный сервис (plan 4 — бывший 3). Amnezia/Outline инфра Timeweb не трогается.

**Known trade-offs we're accepting:**
- Кеш с TTL 60s означает окно до минуты, когда отозванный ключ / заблокированный девайс ещё проходит auth. Для beta-режима с 7 юзерами допустимо; если понадобится мгновенно — позже добавим явный `POST /api/cache/invalidate`, который админка дёргает после revoke.
- Таблица `logs` (654k строк app-логов в БД) переезжает вместе со всем — пустая трата места в Postgres. Чистка — отдельная задача, не в этом плане.
- Тесты в `server/internal/{udp,admin}` уже сломаны в main (см. CLAUDE.md) — мы их не чиним, только делаем так чтобы `server/internal/db/db_test.go` компилился и проходил против Postgres.
- На время между деплоем Aeza и Timeweb: у Aeza новая версия, читает Postgres; у Timeweb ещё старая, читает свой SQLite. Новые пользователи, добавленные через админку в это окно, видны только Aeza. Окно ~10-15 минут, пользователей немного — ок.

## Task 3: Переписать `server/internal/db/db.go` на Postgres

Это самая тяжёлая задача плана — 706 строк SQLite-кода перевести в Postgres. Подход: порт in-place, сохраняя публичный API.

**Files:**
- Modify: `server/internal/db/db.go` (переписать)

**Порт-правила** (применять построчно):
- `database/sql` driver `"sqlite"` → `"pgx"` через `github.com/jackc/pgx/v5/stdlib` (зарегистрируется при blank import).
- Плейсхолдеры `?` → `$1, $2, ...` (нумерация с 1, по порядку появления).
- `INSERT OR IGNORE INTO X VALUES(...)` → `INSERT INTO X VALUES(...) ON CONFLICT DO NOTHING`.
- `INSERT OR REPLACE INTO X (cols) VALUES(...)` → `INSERT INTO X (cols) VALUES(...) ON CONFLICT (conflict_cols) DO UPDATE SET col = EXCLUDED.col, ...`.
- `RETURNING id` работает в обоих — сохранить.
- Bool колонки (`active`, `approved`, `enabled`, `is_primary`): сканить в `bool`, не `int`. Соответственно убрать старые `int`-переменные типа `var active int` → `var active bool`. Где сравнение `active = 1` в WHERE → `active = TRUE`.
- `CURRENT_TIMESTAMP` → `now()` (работает и то и то в PG, но для единообразия — `now()`).
- `datetime('now', '-1 hour')` → `now() - INTERVAL '1 hour'`.
- `strftime('%Y-%m-%d', hour)` → `to_char(hour, 'YYYY-MM-DD')`.
- `strftime('%Y-%m-%dT%H:00:00', ...)` → `to_char(..., 'YYYY-MM-DD"T"HH24":00:00"')`.
- `COALESCE(machine_id, '')` остаётся — работает в PG.
- Весь inline-миграционный код (init schema, `ALTER TABLE devices ADD COLUMN ...`, changelog schema migration) УДАЛЯЕМ. Схему применяет руками админ через `schema.sql` один раз (уже сделано в plan 2 и Task 1). Приложение не создаёт таблицы, не правит схему.
- `sqlite_master` запросы — убрать целиком (ими проверялась старая changelog-схема; схема теперь канонична).
- `cleanOldLogs`: была `DELETE FROM logs WHERE id NOT IN (SELECT id FROM logs ORDER BY id DESC LIMIT 10000)`. Работает в PG as-is. Сохранить.
- `WriteLog`: `INSERT INTO logs ...` без `RETURNING` — параметры `$1, $2`. Сохранить goroutine + 1000-item batching если были (они не были, просто прямой INSERT).
- `Open(path string)` → `Open(url string)`. Параметр становится Postgres URL. Внутри:
  ```go
  sqlDB, err := sql.Open("pgx", url)
  ```
  Вместо `sql.Open("sqlite", path)`.
- `db.SQL() *sql.DB` — остаётся, используется в тестах.

**Хитрые места:**
- `GetOverview()` — агрегатные SELECT'ы. `COUNT(CASE WHEN ...)` работает в PG идентично.
- `GetTrafficByDay` возвращает `[]map[string]interface{}` с днями. Убедиться что `to_char` даёт правильный формат.
- `syncChangelog()` — читает `changelog.json` из файла и делает `INSERT OR IGNORE` на каждую запись. Порт: `INSERT INTO changelog ... ON CONFLICT (id) DO NOTHING`.
- `RecordTraffic` использует `INSERT OR REPLACE` по уникальному индексу `(device_id, hour)`. Порт:
  ```sql
  INSERT INTO traffic_stats (device_id, hour, bytes_in, bytes_out, connections)
  VALUES ($1, $2, $3, $4, $5)
  ON CONFLICT (device_id, hour) DO UPDATE SET
    bytes_in    = traffic_stats.bytes_in    + EXCLUDED.bytes_in,
    bytes_out   = traffic_stats.bytes_out   + EXCLUDED.bytes_out,
    connections = traffic_stats.connections + EXCLUDED.connections
  ```
  **ВАЖНО:** текущая SQLite-реализация под `INSERT OR REPLACE` ПЕРЕЗАПИСЫВАЕТ старые байты новым значением (стирает старое), а не суммирует — читать её код буквально. Порт В PG должен делать ТО ЖЕ, что делает текущий код, не "то что кажется правильным". **Прочитать существующий `RecordTraffic`** и сохранить семантику: если там сейчас суммирование вручную (SELECT then UPDATE) — порт делает то же. Если просто REPLACE — в PG это `DO UPDATE SET bytes_in = EXCLUDED.bytes_in` (перезапись, не суммирование).

## Task 4: Добавить кеш девайсов перед `GetDeviceByKey`

Лёгкая работа — вынести кеширование в отдельный тип, обернуть hot-path метод. Инвалидацию вызывать из admin-мутаций.

**Files:**
- Create: `server/internal/db/devicecache.go`
- Modify: `server/internal/db/db.go` (конструктор DB создаёт кеш; `GetDeviceByKey` смотрит в кеш; `SetDeviceActive`/`DeleteDevice`/`UpdateDeviceVersion`/`SetDeviceMachineID` инвалидируют)

- [ ] **Step 1: Создать devicecache.go**

```go
package db

import (
	"sync"
	"time"
)

const deviceCacheTTL = 60 * time.Second

type cachedDevice struct {
	dev Device
	err error // cache negative lookups too (not found → don't hit DB for 60s on brute-force probes)
	exp time.Time
}

type deviceCache struct {
	mu sync.RWMutex
	m  map[string]cachedDevice
}

func newDeviceCache() *deviceCache {
	return &deviceCache{m: make(map[string]cachedDevice)}
}

func (c *deviceCache) get(key string) (Device, error, bool) {
	c.mu.RLock()
	e, ok := c.m[key]
	c.mu.RUnlock()
	if !ok || time.Now().After(e.exp) {
		return Device{}, nil, false
	}
	return e.dev, e.err, true
}

func (c *deviceCache) put(key string, d Device, err error) {
	c.mu.Lock()
	c.m[key] = cachedDevice{dev: d, err: err, exp: time.Now().Add(deviceCacheTTL)}
	c.mu.Unlock()
}

func (c *deviceCache) invalidate(key string) {
	c.mu.Lock()
	delete(c.m, key)
	c.mu.Unlock()
}

func (c *deviceCache) invalidateAll() {
	c.mu.Lock()
	c.m = make(map[string]cachedDevice)
	c.mu.Unlock()
}
```
